# QQE-1584 Quarkus LangChain4J testing

JIRA link: https://issues.redhat.com/browse/QQE-1584

This is a high-level strategy for testing of langchain4j

## Scope of the testing
@SystemMessage + @UserMessage annotations
RAG functionality
Integration with MCP API

### Optional, but desirable:
@Tool + @RegisterAiService (only supported for OpenAI, so cannot be tested locally)
Observability for tool requests (to avoid a situation, where LLM overwhelms the database with too many requests)

### Quakriverse extension:
There are integration tests modules in the extension, but they contain no tests at the moment.
According to Mario, this is intentional, since they consider upstream lang4chain tests to be enough (but we do not!)

## Automated test development
### Functional testing
1. Add some tests for existing functionality into https://github.com/quarkiverse/quarkus-langchain4j/tree/main/integration-tests/ollama, https://github.com/quarkiverse/quarkus-langchain4j/tree/main/integration-tests/simple-ollama and https://github.com/quarkiverse/quarkus-langchain4j/tree/main/integration-tests/jlama
2. Beware, that these modules use models of different size, and this can affect reproducibility of tests. Some tests may not work on our hardware with some of the models.
3. Investigate, whether it is possible to have RAG with ollama/jlama (it should be: [1][2]), add some new functionality and coverage for it into integration tests (maybe in new module)
4. Add some observability tests.
5. Add ability to change version of Quarkus and quarkus-langchain4j, which is used in integration tests. Adding new property for version in module pom should be enough.
6. MR should target main, the reviewer(s) should be Mario and/or Georgios, since they both agreed, that this is the way to go.
7. Try running the tests on our test servers. If it fails, we may want to try running ollama on a separate machine and connect to it. 
8. Create a new Jenkins job for prod testing on x86_64 in "general" scope and one for ARM_64 in "extended" scope.

In case our hardware is incapable of running existing models, we should consider using some lighter ones, eg DeepSeek ones or quantizied version of existing LLMs.
This may lead to more hallucinations in answers, so we should be very careful with out test cases.

### Integration with OpenShiftAI:
Openshift AI describes itself as "a platform for data scientists and developers of artificial intelligence and machine learning (AI/ML) applications"[3] and mostly focuses on users, who train their own models.
While some press-releases say[4], that Red Hat offers "third party validated models, available on Hugging Face" all examples I was able to find[5] are Python and Jupyter centric.

Since focus of langchain4j is to help users to work with already trained models, I consider OpenShiftAI to be out-of-scope for this effort.

### OpenAI testing [optional]

##### Budget considerations:
We have (on average) one minor release per two months (with general scope testing) or two major releases per year (which include extended scope testing).
For every major release we run extended scope approximately 5-10 times and general scope the same amount of times. For every minor release we run general testing 1 or 2 times.
OpenAI is priced based on amount of tokens, where "token" is something like a root of a word[6].
The cheapest model[7] is gpt-4.1-nano with prices of $0.10 for 1M tokens of input($0.025 for cached input) and $0.40 for 1M token of output.
Workshop uses ~400 token of input. Output can be different, but let's assume 100 tokens per test as an upper limit.
That means, that one million runs of single workshop test will cost us 40 dollars for input and 40 for output, for a total cost of 80 dollars, and one dollar will allow us to do 10 000 runs.
If we put these tests into general testing (so, ~30 runs per year) and adding the debugging and development (100 runs altogether?), which means, that with current prices single dollar would be enough for approximately a century of testing.

# QQE-1584 Quarkus LangChain4J testing

JIRA link: https://issues.redhat.com/browse/QQE-1584

This is a high-level strategy for testing of langchain4j

## Scope of the testing
@SystemMessage + @UserMessage annotations
RAG functionality
Integration with OpenShiftAI tests

### Optional, but desirable:
@Tool + @RegisterAiService (only supported for OpenAI, so cannot be tested locally)
Observability for tool requests (to avoid a situation, where LLM overwhelms the database with too many requests)

### Quakriverse extension:
There are integration tests modules in the extension, but they contain no tests at the moment.
According to Mario, this is intentional, since they consider upstream lang4chain tests to be enough (but we do not!)
Ollama modules can be started on the laptop in dev mode. Jlama ones can be run as well.

## Automated test development
### Functional testing
1. Add some tests for existing functionality into https://github.com/quarkiverse/quarkus-langchain4j/tree/main/integration-tests/ollama, https://github.com/quarkiverse/quarkus-langchain4j/tree/main/integration-tests/simple-ollama and https://github.com/quarkiverse/quarkus-langchain4j/tree/main/integration-tests/jlama
2. Beware, that these modules use models of different size, and this can affect reproducibility of tests. Some tests may not work on our hardware with some of the models.
3. Investigate, whether it is possible to have RAG with ollama/jlama (it should be: [1][2]), add some new functionality and coverage for it into integration tests (maybe in new module)
4. Add some observability tests.
5. Add ability to change version of Quarkus and quarkus-langchain4j, which is used in integration tests. Adding new property for version in module pom should be enough.
6. MR should target main, the reviewer(s) should be Mario and/or Georgios, since they both agreed, that this is the way to go.
7. Try running the tests on our test servers. If it fails, we may want to try running ollama on a separate machine and connect to it. 
8. Create a new Jenkins job for prod testing on x86_64 in "general" scope and one for ARM_64 in "extended" scope.

### Integration with OpenShiftAI:
1. Investigate, which models can be run there, and whether we have access to GPU on our hardware.
2. Unless it is Ollama/Jlama, tests for this model should be added into existing/new module in https://github.com/quarkiverse/quarkus-langchain4j/tree/main/integration-tests/.
3. We need to find a way to deploy and then delete the model/container during the tests' run. Consider moving this functionality to Quarkus QE TS, if this make things easier.
4. Jenkins job for x86_64 should be a part of "general" scope 

### OpenAI testing [optional]
if there is an ability to run tests with OpenAI (ie we have access to credits for testing) we need to:
1. double check tests in https://github.com/quarkiverse/quarkus-langchain4j/tree/main/integration-tests/openai.
2. Add observability tests there, with focus on observability of external calls (these cost real money, so it is important, that users can monitor the usage)
3. Add Jenkins tests, probably in "extended" scope only.
